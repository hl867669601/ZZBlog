IO模型

发表于 2022-11-17 更新于 2022-11-18 分类于 IO

五大IO模型

一次TCP通信的过程如图所示：

首先A应用要把发送的数据放到TCP缓冲区中、然后通过TCP/IP协议发送到对方的TCP接受缓冲区。然后应用B再从缓冲区拿数据。

而应用间的数据通信是间断性的，当应用向TCP缓冲区发出读取请求时，针对IO模型的不同，操作系统会给出不同的操作。

名词解释：

  • 阻塞/非阻塞关注的是程序在等待调用结果（消息，返回值）时的状态。主要关注的是线程的状态会否转变为等待。

阻塞：阻塞调用是指调用结果返回之前，当前线程会被挂起。调用线程只有在得到结果之后才会返回。

非阻塞：非阻塞调用指在不能立刻得到结果之前，该调用不会阻塞当前线程。

  • 同步/异步关注的是消息通信机制 (synchronous communication/ asynchronous communication) 。

同步：所谓同步，就是在发出一个调用时，在没有得到结果之前， 该调用就不返回。

异步：则是相反，调用在发出之后，这个调用就直接返回了，所以没有返回结果

阻塞式IO

当，系统会等待数据到达，并把数据复制到用户内存空间时才返回。用户进程从发起请求到内核返回数据全程阻塞。

  1. 应用进程发起recvfrom读取数据（应用阻塞）。
  2. 将数据从内核复制到应用空间。
  3. 返回。

非阻塞IO

如果缓冲区没有数据：

  1. 应用进程向内核发起recvfrom读取数据（不阻塞）。
  2. 立刻返回错误码。

如果缓冲区有数据：

  1. 应用进程向内核发起recvfrom读取数据（不阻塞）。
  2. 将数据从内核空间拷贝到用户空间
  3. 返回。

异步I/O

  1. 发起aio_read。
  2. 立即返回，线程可以执行别的。
  3. 当数据到了，并拷贝到用户空间了执行回调。
  • 一个非阻塞I/O 系统调用 read() 操作立即返回的是任何可以立即拿到的数据， 可以是完整的结果， 也可以是不完整的结果， 还可以是一个空值。
  • 而异步I/O系统调用 read（）结果必须是完整的， 但是这个操作完成的通知可以延迟到将来的一个时间点。

I/O复用

如果并发环境下，每个应用都要及时读取到数据，这些线程就会不断的发送recvfrom请求，就会导致大量资源浪费。

然后就出现了一种只用一个线程去检查每个IO请求的数据状态询问。然后再对数据准备就绪的IO连接分配线程去读取数据。

总结：复用IO的基本思路就是通过slect或poll、epoll 来监控多fd ，来达到不必为每个fd创建一个对应的监控线程，从而减少线程资源创建的目的。

信号驱动I/O

复用IO模型解决了一个线程可以监控多个fd的问题，但是select是采用轮询的方式来监控多个fd的，通过不断的轮询fd的可读状态来知道是否就可读的数据，但这样的不断轮询也会导致性能浪费。

应用进程使用 sigaction 系统调用，内核立即返回，应用进程可以继续执行，也就是说等待数据阶段应用进程是非阻塞的。内核在数据到达时向应用进程发送 SIGIO 信号，应用进程收到之后在信号处理程序中调用 recvfrom 将数据从内核复制到应用进程中。

SELECT POLL EPOLL

SELECT

  1. 使用copy_from_user从用户空间拷贝fd_set到内核空间

  2. 遍历所有fd，如果没有一个可读写的fd(文件描述符)，就会睡眠一定时间。超时或被唤醒后重新遍历。

  3. 如果有可读写的就把把fd_set从内核空间拷贝到用户空间。

  4. 用户进程遍历fd_set，处理可读写的IO

    缺点：

    • 每次调用select，都需要把fd集合从用户态拷贝到内核态，这个开销在fd很多时会很大
    • 同时每次调用select都需要在内核遍历传递进来的所有fd，这个开销在fd很多时也很大
    • select支持的文件描述符数量太小了，默认是1024

POLL

SELECT的链表版。（没有最大文件描述符数量限制）

EPOLL

事件驱动

有两种模式：

LT(水平触发): 默认，只要该fd还有数据可读，每次epoll_wait都会返回它的事件。

ET(边缘触发): 只会返回一次，除非产生第二次读写事件。

  1. epoll_create：

​ 调用epoll_create时，内核会创建一个eventpoll内核对象。

  1. epoll_ctl

    调用epoll_ctl后会分配一个红黑树节点对象epitem，并插入到红黑树中

  2. epoll_wait
    epoll_wait 做的事情不复杂，当它被调用时它观察 eventpoll->rdllist 链表里有没有数据即可。有数据就返回，没有数据就创建一个等待队列项，将其添加到 eventpoll 的等待队列上，然后把自己阻塞掉就完事。

  3. Socket数据到时，找到Socket对应的红黑树节点，并把节点添加到就序列表中，然后唤醒在epoll阻塞的线程。

零拷贝

DMA

传统IO：

  1. CPU发出指令给需要数据传输的磁盘控制器，然后返回

  2. 磁盘控制器收到后，会准备数据，将数据放到磁盘控制器的内部缓冲区，然后产生中断

  3. CPU中断后再将缓冲区的数据一个字节一个字节的读入寄存器，然后再把寄存去的数据写入到内存中。全程CPU参与搬运过程。

DMAIO:

  1. cpu发送IO请求到DMA.

  2. DMA负责数据的搬运，搬运到内核空间，发送中断请求给CPU

  3. CPU收到后，知道数据准备好了，就将数据拷贝到用户空间并返回。

文件传输

传统文件传输

read+write

4次拷贝：

  1. 第一次拷贝，把磁盘的数据拷贝到内核缓冲区（DMA）。

  2. 第二次拷贝，把内核缓冲区的数据拷贝到用户缓冲区

  3. 第三次拷贝，把用户缓冲区的数据拷贝到socket缓冲区

  4. 第四次拷贝，把scoket缓冲区的数据拷贝到网卡

4次上下文切换：

  1. 用户发起read（从用户态转切换内核态）。

  2. 内核把数据拷贝到用户缓冲区后返回（从内核态切换到用户态）

  3. 用户发起write（从用户态切换到内核态）

  4. 内核发送完后返回（从内核态切换到用户态）

mmap+write

  1. 第一次拷贝，应用进程调用mmap()后，DMA会把磁盘数据拷贝到内核缓冲区，此时内核和用户共享这个缓冲区。
  2. 第二次拷贝，用户调用write, cpu将缓冲区的数据拷贝到socket缓冲区
  3. 第三次拷贝，DMA将socket缓冲区数据拷贝到网卡。

mmap+write减少了一次数据拷贝，但还是4次上下文切换(因为还是两次系统调用)。

sendfile

  1. 第一次拷贝，DMA直接将磁盘数据拷贝到内核缓冲区

  2. 第二次拷贝，cpu将内核缓冲区数据拷贝到socket缓冲区

  3. 第三次拷贝，DMA将socket缓冲区数据拷贝到网卡

    三次数据拷贝，两次上下文切换（只有一次系统调用）

如果支持SG-DMA的话可以直接将内核缓存区中的数据拷贝到网卡缓冲区
